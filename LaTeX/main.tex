\begin{abstract}
    This paper presents Curia, a campus event search system that converts natural language queries into structured temporal and semantic constraints for downstream retrieval. Given a query such as ``Show me concerts next Saturday evening,'' the system extracts date ranges, time-of-day bounds, and semantically expanded keyword sets, then scores and ranks events from a live corpus of UNL events. Curia implements two interchangeable inference paths: a remote path using the Gemini API (\texttt{gemma-3-27b-it}, 27B parameters) and a fully local path using \texttt{Qwen2.5-0.5B-Instruct} with few-shot in-context prompting via chat templates. Both paths are evaluated on a rigorous benchmark of 72 temporal queries drawn from real event data, with ground-truth date ranges, time bounds, keywords, and relevant URLs labeled per case. The Gemini path achieves 100\% temporal parse success, 100\% date and time exact-match accuracy, and a Mean Reciprocal Rank (MRR) of 0.650. The local Qwen2.5-0.5B path achieves 23.6\% parse success and 19.4\% date exact-match at 6.9\,s mean CPU latency, revealing a clear capability ceiling at sub-1B scale. A lightweight hybrid retrieval module (\texttt{all-MiniLM-L6-v2}, 22M parameters) augments keyword expansion for both paths at under 50\,ms. We further report on four additional local model candidates—\texttt{Qwen2.5-1.5B}, \texttt{TinyLlama-1.1B}, \texttt{flan-t5-base}, and \texttt{LaMini-Flan-T5-248M}—all of which fail to produce structured output on the benchmark, establishing the difficulty of the task at small scale.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Event search systems must bridge the gap between natural language user intent and structured database filters. A user typing ``free food next Thursday after 3pm'' has simultaneously specified a topic (food), a relative date anchor (next Thursday), and a time lower bound (15:00). Most retrieval systems handle keyword matching well but struggle with temporal reasoning, requiring users to manually select dates from a calendar widget.

Existing approaches to temporal query grounding tend toward one of two extremes. Rule-based systems (e.g., \texttt{dateparser}~\cite{dateparser}) handle explicit date strings and a small set of relative expressions but fail on implicit or compound references. Large language models handle a broad range of expressions but require API access, incur significant latency, and cannot run offline. Sequence labeling approaches based on TIMEX3~\cite{verhagen2007semeval} annotation, while well-studied in the news domain, do not transfer cleanly to query-side temporal resolution against a present-day reference point.

Curia takes a pragmatic middle path. The system is built around two inference modes that share the same retrieval pipeline and can be swapped via environment variable:

\begin{itemize}
    \item \textbf{Gemini path} — A single API call to \texttt{gemma-3-27b-it} jointly extracts temporal constraints and expands query keywords. Reliable and accurate but requires network access and API credentials.
    \item \textbf{Local path} — \texttt{Qwen2.5-0.5B-Instruct} receives few-shot chat-template prompts independently for temporal extraction and keyword expansion, followed by \texttt{dateparser} and a MiniLM semantic retrieval fallback. Runs entirely on consumer CPU hardware.
\end{itemize}

Both paths produce the same structured output format (JSON with \texttt{date\_from}, \texttt{date\_to}, \texttt{time\_from}, \texttt{time\_to}, and an expanded keyword list) consumed by a shared scoring and ranking layer in the Express backend.

The contributions of this paper are:
\begin{itemize}
    \item A two-stage query grounding pipeline that separates temporal extraction from semantic keyword expansion, with pluggable LLM backends
    \item A rigorous benchmark dataset of 72 temporal queries derived from real UNL events, with ground-truth temporal bounds, keywords, and relevant event URLs
    \item An empirical comparison of six model candidates ranging from 248M to 27B parameters on parse success, temporal accuracy, keyword quality, retrieval MRR, and CPU latency
    \item A hybrid retrieval module that augments LLM keyword output with dense nearest-neighbor lookup over a pre-embedded 384-term domain vocabulary
\end{itemize}

\section{Related Work}
\label{sec:related}

\subsection{Temporal Information Extraction}

Temporal extraction is a well-studied problem in NLP. The TIMEX3 annotation scheme~\cite{verhagen2007semeval} defines structured representations for dates, times, durations, and sets. Systems such as HeidelTime~\cite{strotgen2010heideltime} and SUTime~\cite{chang2012sutime} use rule-based approaches to normalize temporal expressions relative to a document creation time (DCT). Neural approaches fine-tuned on TimeBank~\cite{pustejovsky2003timebank} improve recall on news text but inherit the document-centric assumption: temporal expressions in news articles are anchored to the article's publication date, not to the user's present moment. As we demonstrate in Section~\ref{sec:experiments}, a model trained for TIMEX3 tagging (\texttt{temporal\_tagger\_roberta2roberta}) fails completely when applied to query-side resolution against a live date context.

\subsection{Few-Shot and In-Context Learning}

In-context learning (ICL)~\cite{brown2020language} enables language models to perform new tasks given only a handful of demonstration examples in the prompt, without any gradient updates. Chat-template prompting—where examples are formatted as alternating \texttt{user}/\texttt{assistant} turns—improves instruction-following in decoder-only models~\cite{touvron2023llama}. We use this approach with \texttt{Qwen2.5-0.5B-Instruct}, providing 4 exemplars for temporal extraction and 3 exemplars for keyword expansion. The model weights are never updated; all adaptation is in-context.

\subsection{Query Understanding and Structured Extraction}

Semantic parsing systems~\cite{berant2013semantic} convert natural language into structured representations (SQL, SPARQL, or domain-specific JSON). For event retrieval, the required output is simpler than general-purpose semantic parsing: a flat JSON object with date/time fields and a keyword list. End-to-end generation approaches that directly produce structured output have become practical with small instruction-tuned models, provided the output schema is simple and the model is given explicit format constraints in the prompt.

\subsection{Dense Retrieval and Hybrid Search}

Dense retrieval models~\cite{karpukhin2020dense} encode queries and documents into shared embedding spaces, enabling semantic matching beyond keyword overlap. For keyword expansion specifically, embedding-based nearest-neighbor lookup over a domain vocabulary~\cite{reimers2019sentencebert} is a lightweight technique to supplement LLM output without requiring a separate retrieval-augmented generation (RAG) pipeline. Our MiniLM hybrid module follows this pattern, using cosine similarity over a pre-computed 384-term embedding corpus to extend the keyword set produced by the LLM.

\subsection{Hybrid Rule and Learning-Based Systems}

Combining neural models with rule-based fallbacks improves robustness on out-of-distribution inputs~\cite{namazifar2021language}. For temporal grounding in particular, \texttt{dateparser} provides reliable coverage of explicit date strings (``April 25'', ``2026-04-25'') and common relative expressions (``tomorrow'', ``next week'') that deterministic code handles better than a small neural model. Our system applies \texttt{dateparser} whenever the LLM's JSON output is malformed or produces implausible date values.

\section{Methodology}
\label{sec:method}

\subsection{System Overview}

Curia consists of three components: (1) a FastAPI scraper service that collects and refreshes UNL event data, (2) a query grounding module that converts natural language queries into structured constraints via one of two LLM paths, and (3) an Express backend that applies those constraints to score and rank events from a SQLite store.

Figure~\ref{fig:architecture} illustrates the overall data flow. When a user submits a search query, the Express backend forwards it to the FastAPI service. The FastAPI service runs the query through the active LLM path (Gemini or local), returns expanded keywords and temporal bounds as JSON, and the backend applies them to rank all events.

\subsection{Task Definition}

Given a natural language query $q$, the system produces:
\begin{itemize}
    \item \texttt{date\_from}, \texttt{date\_to}: ISO 8601 date bounds (YYYY-MM-DD), or null if no date constraint is present
    \item \texttt{time\_from}, \texttt{time\_to}: 24-hour time bounds (HH:MM), or null if no time constraint is present
    \item \texttt{keywords}: a list of semantically expanded search terms
\end{itemize}

For example, the query ``morning concerts next weekend'' should produce:
\begin{lstlisting}[language={}]
{
  "date_from": "2026-05-02",
  "date_to":   "2026-05-03",
  "time_from": "06:00",
  "time_to":   "12:00",
  "keywords":  ["concert", "music",
                "performance", "live music",
                "band", "show"]
}
\end{lstlisting}

\subsection{Gemini Remote Path}

The default production path sends a single structured prompt to \texttt{gemma-3-27b-it} via the \texttt{google-genai} SDK. The prompt includes the current date (injected at runtime), the user query, and instructions to return a JSON object with temporal fields and an expanded keyword list. Because the model is large (27B parameters) and instruction-tuned, it reliably produces parseable JSON and handles compound temporal expressions such as ``two weekends from now'' and ``Thursday afternoon'' without additional scaffolding.

This path requires a valid \texttt{GEMINI\_API\_KEY} and network access. Mean latency is approximately 7.9\,s over the benchmark dataset (p95 = 8.7\,s).

\subsection{Local Two-Stage Path}
\label{sec:local}

The local path (\texttt{USE\_LOCAL\_MODELS=true}) runs \texttt{Qwen2.5-0.5B-Instruct} (500M parameters, $\approx$954\,MB weights) on CPU using the HuggingFace \texttt{transformers} library. Two independent prompts are issued sequentially:

\paragraph{Stage 1 — Temporal Extraction}
The model is prompted with 4 few-shot exemplars formatted as \texttt{user}/\texttt{assistant} chat turns. Each exemplar pairs a natural language query with the expected JSON temporal output. A ``Reasoning: \ldots'' prefix (chain-of-thought) is prepended to each assistant turn to encourage step-by-step date arithmetic before the final JSON. The current date is injected into the system message so the model can resolve relative expressions.

\paragraph{Stage 2 — Keyword Expansion}
A separate prompt provides 3 exemplars mapping queries to 20--25 expanded keyword lists. The model generates a JSON array of terms. The output is merged with results from the MiniLM hybrid retrieval module (Section~\ref{sec:minilm}).

Both stages use greedy decoding (\texttt{do\_sample=False}) with a maximum of 512 new tokens. If either stage produces malformed JSON or the date parse fails, the system falls back to \texttt{dateparser} for temporal extraction and raw keyword splitting for expansion.

\paragraph{Fallback Chain}
The full fallback hierarchy is:
\begin{enumerate}
    \item LLM JSON output (parsed and validated)
    \item \texttt{dateparser} for date fields; time-of-day keyword lookup for time fields
    \item Null bounds (no temporal constraint applied)
\end{enumerate}

\subsection{MiniLM Hybrid Retrieval}
\label{sec:minilm}

\texttt{all-MiniLM-L6-v2} (22M parameters, $\approx$90\,MB) encodes the query and a pre-embedded vocabulary of 384 domain terms drawn from \texttt{backend/keywords.js}. At query time, the top-$k$ most similar vocabulary terms (by cosine similarity) are appended to the LLM's keyword list. The embedding corpus is computed once at startup and cached in memory, so retrieval adds under 50\,ms per query. This module is active for both the Gemini and local paths.

\subsection{Event Scoring}

The Express backend ranks events using a weighted field scoring scheme:

\begin{equation}
\text{score}(e, Q) = \sum_{t \in Q} w_f \cdot \mathbf{1}[t \in \text{field}_f(e)] \cdot \text{pos}(t, Q)
\end{equation}

where $w_f$ is the field weight (name: 4, venue/category: 2, description/tags: 1), and $\text{pos}(t, Q)$ is a position bonus that gives higher weight to terms appearing earlier in the query. Date and time filters are applied as hard constraints before scoring: events outside the extracted temporal bounds are excluded entirely.

\section{Experiments}
\label{sec:experiments}

\subsection{Benchmark Dataset}

We construct a rigorous evaluation dataset of 72 temporal queries drawn from real UNL events (collected via the Curia scraper). Each case includes:
\begin{itemize}
    \item A natural language query (e.g., ``events on 2026-03-31 about alcohol drug'')
    \item Ground-truth \texttt{date\_from}, \texttt{date\_to}, \texttt{time\_from}, \texttt{time\_to}
    \item A set of expected keyword labels
    \item One or more relevant event URLs from \texttt{scraped/events.json}
    \item Categorical tags (\texttt{date}, \texttt{time}, \texttt{topic}, \texttt{temporal})
\end{itemize}

Cases are organized into variant clusters (A/B/C per topic) that test different phrasings of the same underlying temporal constraint. The dataset is stored at \path{testing/benchmark/datasets/queries_rigorous_temporal.json}.

\subsection{Metrics}

We report the following metrics:

\begin{itemize}
    \item \textbf{Parse Success Rate}: Fraction of queries where the model returns valid, parseable JSON
    \item \textbf{Date Exact Rate}: Fraction of queries where both \texttt{date\_from} and \texttt{date\_to} exactly match ground truth
    \item \textbf{Time Exact Rate}: Fraction of queries with ground-truth time bounds where both time fields exactly match
    \item \textbf{Keyword F1}: Token-level F1 between predicted and expected keyword sets (Jaccard-based)
    \item \textbf{Top-$k$ Hit Rate}: Fraction of queries where the ground-truth event URL appears in the top-$k$ retrieved results
    \item \textbf{MRR}: Mean Reciprocal Rank of the ground-truth event URL across all retrieved results
    \item \textbf{Latency}: Mean and p95 wall-clock inference time in milliseconds (CPU, 4-core)
\end{itemize}

\subsection{Ablation: Model Scale and Capability Ceiling}
\label{sec:ablation}

In a separate 8-query exploratory evaluation (reported in Table~\ref{table:scale}), we compared \texttt{Qwen2.5-0.5B} and \texttt{Qwen2.5-1.5B} directly on the same queries. The 1.5B model achieved 3/7 date matches and 2/2 time-of-day matches, versus 0.5B's partial success. However, CPU latency for the 1.5B model was $\approx$129\,s per query—14$\times$ slower than Gemini and impractical for interactive use. These results suggest that the 0.5B--1B parameter range represents a capability ceiling for few-shot temporal arithmetic on consumer CPU hardware, and that models in this range would require GPU acceleration or aggressive quantization (GGUF/int4) to be viable in production.

\subsection{Abandoned Candidates}

Two additional model families were evaluated and abandoned before the rigorous benchmark:

\paragraph{TIMEX3 Sequence Labeler (\texttt{temporal\_tagger\_roberta2roberta})}
This 250M encoder-decoder model was fine-tuned for TIMEX3 annotation of news text. Applied to query-side temporal resolution, it produced bare \texttt{<TIMEX3 value="">} tags anchored to document context rather than the user's present date, inflating every query to a full-year range (2026-01-01 to 2026-12-31) or hallucinating year values (e.g., 1998-W44). Result: 0/8 date range matches. The domain mismatch between document-centric TIMEX3 tagging and query-side resolution is fundamental; this approach was discontinued.

\paragraph{Zero-Shot Seq2Seq (\texttt{flan-t5-base})}
Zero-shot prompting of \texttt{flan-t5-base} for JSON output produced alphabetic enumeration (``a, b, c, d\ldots'') or input token repetition. The model has no chat template; structured JSON generation at 250M parameters without fine-tuning is unreliable. This model achieves 100\% parse success in the benchmark (the seq2seq output is syntactically well-formed) but 0\% accuracy on every substantive metric.

\subsection{Error Analysis}

Failure modes for \texttt{Qwen2.5-0.5B-Instruct} on the rigorous benchmark fall into three categories:

\begin{itemize}
    \item \textbf{JSON generation errors} (76.4\% of cases): Unterminated strings, truncated output, or extra text after the closing brace. These trigger the fallback chain.
    \item \textbf{Week boundary confusion}: ``this week'' is anchored to the current weekday rather than the Monday--Sunday span.
    \item \textbf{Day-of-week arithmetic}: ``next Thursday'' returns today's date when today is a weekday, indicating the model does not reliably advance to the next occurrence.
\end{itemize}

The Gemini model has no observed failures on the 72-case benchmark, consistent with its 100\% parse success and temporal accuracy scores.

\section{Conclusion}
\label{sec:conclusion}

We present Curia, a campus event search system with a two-path natural language query grounding pipeline. The system demonstrates that a large remote model (Gemini \texttt{gemma-3-27b-it}) can achieve perfect temporal extraction and strong retrieval (MRR 0.650) on real-world campus event queries, while a small local model (\texttt{Qwen2.5-0.5B-Instruct}) reaches only partial success at 23.6\% parse reliability despite identical few-shot prompting.

The key findings from our evaluation:

\begin{itemize}
    \item \textbf{Scale matters for structured temporal output}: No model below 27B parameters achieves consistent JSON generation with correct temporal arithmetic on the rigorous benchmark. The 0.5B model achieves partial parse success only.
    \item \textbf{Domain transfer is difficult for TIMEX3 models}: Sequence labelers trained on news documents do not transfer to query-side temporal resolution against a present-day reference.
    \item \textbf{Hybrid retrieval adds minimal overhead}: MiniLM semantic retrieval augments keyword output at under 50\,ms, making it a practical addition to both inference paths.
    \item \textbf{Fallback chains improve robustness}: \texttt{dateparser} recovers explicit date strings that the local model fails to parse, enabling graceful degradation on explicit-date queries even when the LLM fails.
    \item \textbf{The quality-latency tradeoff is non-linear}: The 1.5B model is 14$\times$ slower than the API path yet far less accurate, indicating that consumer CPU is not a viable target for models above $\approx$500M parameters without quantization.
\end{itemize}

\subsection{Future Work}

\begin{itemize}
    \item \textbf{Quantized local models}: Evaluate GGUF int4/int8 variants of Qwen2.5-1.5B and Qwen2.5-3B to improve the local accuracy-latency frontier
    \item \textbf{Confidence-gated fallback}: Use the model's token log-probabilities to decide when to trust LLM output vs.\ fall back to rules
    \item \textbf{Recurring event queries}: Extend temporal bounds extraction to handle recurring expressions (``every Monday'', ``weekly seminars'')
    \item \textbf{Retrieval-augmented grounding}: Provide relevant events from the corpus as context, allowing the model to verify its temporal predictions against actual event dates
    \item \textbf{Expanded benchmark}: Add robustness (typo) and cross-domain query variants to the evaluation dataset
\end{itemize}
