\begin{abstract}
Curia is a campus event search system for University of Nebraska--Lincoln that parses natural-language queries into keywords and structured temporal bounds. This paper studies how far the underlying language model can be reduced while preserving keyword expansion and temporal grounding under tight CPU latency and memory budgets.

We evaluate seven model candidates---a remote 27B reference (Gemini), a local 8B baseline (Ollama llama3), and five Hugging Face models (248M--1.5B)---on a rigorous benchmark of 72 labeled temporal queries. The remote Gemini model achieves 91.7\% parse success and 91.7\% date exact-match accuracy. Among local models, Ollama llama3 (8B) is the strongest candidate with 26.4\% date and 35.4\% time exact-match accuracy at 3.5\,s mean CPU latency. Smaller models degrade sharply: Qwen 0.5B reaches only 36.1\% date accuracy; sub-1B encoder-decoder models (flan-t5-base, LaMini) achieve zero temporal accuracy despite high parse success. Results reveal asymmetric degradation: keyword expansion is more robust to capacity reduction than temporal grounding, with keyword F1 retaining 37\% of Gemini's performance at 8B parameters while temporal accuracy falls to 26\%. This suggests a structured-output ceiling exists below 1B parameters for temporal reasoning on consumer CPU, and that alternative architectural choices (encoder-decoder vs.\ decoder-only) do not overcome this capacity barrier.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Curia's natural language search system depends on two query-understanding capabilities: extracting and expanding intent-bearing keywords for retrieval, and grounding temporal expressions into structured date and time bounds. A user typing ``free food next Thursday after 3pm'' has simultaneously specified a topic, a relative date anchor, and a time lower bound. Converting this free-form input into structured filters drives retrieval quality.

In practice, the search must operate under tight memory and latency budgets, which pushes deployment toward smaller models. This raises a central design tension: how far can the underlying model be reduced before these extraction behaviors degrade enough to harm retrieval? Prior work on neural scaling laws~\cite{kaplan2020scaling,hoffmann2022training} suggests that capabilities can decline unevenly as capacity decreases, and the effect is often amplified when the task requires reliable structured output.

This paper studies that tension directly in Curia, a campus event search system for UNL. Rather than targeting architectural novelty, we measure how model size affects the two core tasks that drive retrieval: keyword expansion and temporal grounding. We focus on four research questions:
\begin{enumerate}
    \item Does the gap between the two tasks, temporal grounding and keyword expansion, begin to diverge as model size decreases?
    \item At what point can our structured prompt no longer produce usable results?
    \item How stable are these results, and does the stability substantively change as model size decreases?
    \item What model size offers the best tradeoff among stability, accuracy, and latency?
\end{enumerate}

Our experiments show asymmetric degradation. Keyword expansion remains partially functional at 500M parameters, while temporal structured output collapses entirely below that size. A sub-1B structured-output ceiling appears where models either time out or produce malformed JSON on our 72-query rigorous benchmark.

This paper makes three contributions:
\begin{itemize}
    \item \textbf{Task-specific size analysis}: An empirical comparison of seven model candidates (248M to 27B) on keyword expansion and temporal grounding within the same event-search pipeline, evaluated on a benchmark of 72 labeled temporal queries.
    \item \textbf{Failure characterization}: Documentation of how and where temporal extraction breaks down as model capacity decreases, showing that structured-output correctness is unrecoverable below 1B parameters through prompt engineering or fallback heuristics alone.
    \item \textbf{Deployment guidance}: Evidence that keyword expansion is more size-resilient than temporal grounding, with practical implications for feature-selection and model-selection at different deployment scales.
\end{itemize}

\section{Related Work}
\label{sec:related}

Our work sits at the intersection of neural scaling, temporal information extraction, in-context learning, and small-model deployment. We review the most relevant prior work in each area below.

\subsection{Neural Scaling Laws}

Kaplan et al.~\cite{kaplan2020scaling} showed that language model loss scales as a power law with model size, dataset size, and compute across seven orders of magnitude. Hoffmann et al.~\cite{hoffmann2022training} refined this, finding that many widely-used models are undertrained relative to their parameter count and that tokens and parameters should be scaled together. Both results bear directly on our study: if performance degrades predictably with size, we should be able to identify a parameter range where structured extraction breaks down---and our experiments show that range is sharper than a smooth power law would suggest.

\subsection{Temporal Extraction and Hybrid Retrieval}

HeidelTime~\cite{strotgen2010heideltime} is a rule-based system for extracting and normalizing temporal expressions using hand-crafted grammars. It works well on standard patterns but needs domain-specific tuning for new text types. We use \texttt{dateparser} in a similar capacity as a fallback: when the LLM returns malformed JSON, it tries to recover any parseable date strings. The catch is that it only fires on parse failures---when the model returns valid JSON with wrong dates, which is the main failure we document, the fallback never gets a chance to run. For keyword expansion, Reimers and Gurevych~\cite{reimers2019sentencebert} showed that Siamese BERT networks produce dense sentence embeddings well-suited for semantic similarity search. We use \texttt{all-MiniLM-L6-v2}, a compact Sentence-BERT model, to retrieve semantically related domain terms and append them to the LLM's keyword output at under 50\,ms per query.

\subsection{Few-Shot and In-Context Learning}

Brown et al.~\cite{brown2020language} showed that large language models can follow new task specifications from a handful of in-context examples alone, no weight updates needed, and that this ability improves with scale. We rely on this throughout: all models receive the same in-context exemplars (4 temporal, 3 keyword) under an identical prompt contract. Our results quantify the scale dependency: at 27B (Gemini), in-context learning achieves 91.7\% date accuracy; at 8B (Ollama llama3), 26.4\%; at 1B and below, temporal accuracy collapses to near-zero except for specific architecture-prompt combinations. This suggests that in-context temporal reasoning has a hard capacity floor around 1B parameters under CPU execution, aligning with prior observations about structured-output brittleness in smaller models.

\subsection{Small and Specialized Language Models}

Several groups have looked at whether small models are viable substitutes for large ones in narrow domains. Grangier et al.~\cite{grangier2024small} propose resampling pretraining data toward a target domain using importance sampling, alongside a projected-network architecture that compresses a large model into a smaller one at inference time. Belcak et al.~\cite{belcak2025slm} make the broader argument that SLMs are a natural fit for the repetitive, single-purpose subtasks that show up in production agentic pipelines. Fan et al.~\cite{fan2025smallmighty} report that sub-3B models can compete on time-series forecasting when given descriptive statistical prompts and an adaptive fusion layer. These papers make a reasonable case that small models are worth taking seriously for specialized tasks. We test that idea on structured temporal grounding and find the gap to the 27B reference model is not closeable in the sub-2B range under our evaluation conditions.

\section{Methodology}
\label{sec:method}

\subsection{Evaluation Design}

The benchmark is structured as a fixed evaluation matrix over a predefined model set and dataset presets. The model matrix includes seven candidates: a Gemini reference (27B remote), a local Ollama baseline (8B), and five Hugging Face models (248M--1.5B). The primary evaluation dataset is \texttt{rigorous-temporal}, a 72-query benchmark focused on temporal grounding with labeled date/time bounds. The \texttt{all-events} preset covering the full event catalog exceeds 10 hours on consumer machines and is excluded to prioritize reproducibility. A secondary \texttt{rigorous-robustness} dataset is available for perturbation testing but is not included in the main results reported here.

The protocol keeps the inference backend and execution mode explicit. All runs use the local Hugging Face backend on CUDA GPU to ensure hardware-independent latency comparisons. The benchmark measures the interaction between model capacity and the Curia query-understanding pipeline under a stable test harness, permitting clean attribution of performance differences to model scale rather than implementation variation.

\subsection{Task Outputs}

Given a natural language query $q$, Curia produces two outputs:
\begin{itemize}
    \item \textbf{Keywords}: An expanded set of intent-bearing terms for retrieval matching.
    \item \textbf{Temporal bounds}: A JSON object with four fields — \texttt{date\_from}, \texttt{date\_to}, \texttt{time\_from}, \texttt{time\_to} — or \texttt{null} when no constraint is present.
\end{itemize}

For example, ``morning concerts next weekend'' should yield:
\begin{lstlisting}[language={}]
{
  "date_from": "2026-05-02",
  "date_to":   "2026-05-03",
  "time_from": "06:00",
  "time_to":   "12:00",
  "keywords":  ["concert", "music",
                "performance", "band", "show"]
}
\end{lstlisting}

\subsection{Scoring Framework}

The correctness suite summarizes each model with task-level extraction metrics. For a query $q$, the benchmark records whether the model returns parseable JSON, whether the date and time fields match the labeled targets, and whether the predicted keywords align with the expected keyword set. The summary metrics reported in the paper are parse success rate, exact date match, exact or partial time match depending on the case label, and keyword F1 scores. All metrics are aggregated over the full 72-query dataset.

Let $s_{\max}$ denote the Gemini reference model and $s$ any smaller candidate. We normalize each task score to its value at $s_{\max}$:
\[
\hat{F1}_{\mathrm{kw}}(s)=\frac{F1_{\mathrm{kw}}(s)}{F1_{\mathrm{kw}}(s_{\max})}, \quad
\hat{EM}_{\mathrm{temp}}(s)=\frac{EM_{\mathrm{temp}}(s)}{EM_{\mathrm{temp}}(s_{\max})}.
\]

Performance divergence is then:
\[
D_{\mathrm{perf}}(s)=\hat{F1}_{\mathrm{kw}}(s)-\hat{EM}_{\mathrm{temp}}(s).
\]

Positive $D_{\mathrm{perf}}(s)$ means keyword expansion is retained better than temporal extraction at size $s$.

\subsection{Inference Paths}

\paragraph{Gemini Remote Path (27B baseline)}
The reference path sends a single structured prompt to \texttt{gemma-3-27b-it} via the \texttt{google-genai} SDK. The prompt includes the current date and requests a JSON object containing temporal fields, keyword expansion, and retrieval-oriented output. This path serves as the $s_{\max}$ reference for divergence normalization and for the quality-versus-latency comparisons reported in the evaluation.

\paragraph{Local Model Path}
The local candidates execute through the Hugging Face backend on CPU by default, with optional device overrides for CUDA or other supported runtimes. Each local model is evaluated with the same structured prompt contract, the same top-level output schema, and the same decoding settings so that differences in output quality are attributable to model capacity rather than prompt redesign. The benchmark runner preserves a fixed prompt interface across all local candidates; the only variation is the model identity and the execution backend.

For the latency-stability suite, the runner repeats each query three times and summarizes within-model variability using the standard deviation of parse success, keyword F1, and latency p95 across repeated runs. This separates sensitivity to model size from run-to-run jitter and allows the paper to report stability as a first-class property rather than an anecdotal observation.

\paragraph{Controlled Fallback Layer}
The benchmark records model outputs before aggregation and does not require per-model recovery heuristics to pass a gate. For malformed or incomplete outputs, the evaluator treats parse failure, missing temporal fields, and empty keyword lists as part of the observed model behavior. This keeps the measured outcomes aligned with the actual benchmark artifacts and prevents hidden post-processing from masking failures in the reported metrics.

\subsection{MiniLM Hybrid Retrieval}

\texttt{all-MiniLM-L6-v2} (22M parameters) is used as a retrieval augmentation component in the Curia system, but it is not part of the fixed benchmark matrix. Its role is to broaden keyword coverage in the deployed pipeline; the benchmark itself reports the behavior of the language-model candidates under a common evaluation harness, with retrieval treated as downstream infrastructure.

\subsection{Event Scoring}

The backend ranks events by weighted field score:
\[
\text{score}(e, Q) = \sum_{t \in Q} w_f \cdot \mathbf{1}[t \in \text{field}_f(e)] \cdot \text{pos}(t, Q)
\]
where $w_f \in \{4, 2, 1\}$ for name, venue/category, and description/tags respectively, and $\text{pos}(t,Q)$ is a position bonus giving higher weight to terms that appear earlier in the query. Temporal bounds are applied as hard pre-filters before scoring.

\subsection{Model Selection Rationale}

The benchmark model set was selected to probe the parameter-performance frontier at key intervals and to include both instruction-tuned decoder-only and encoder-decoder architectures. \texttt{gemma-3-27b-it} serves as the remote reference model---a production-grade instruction-tuned 27B decoder-only model via the Gemini API that reliably solves both tasks. \texttt{llama3:latest} (8B) is included as a local baseline available via Ollama, bridging between the Gemini scale and smaller local candidates while remaining practical for CPU inference.

The local decoder-only candidates span the sub-1B region: \texttt{Qwen2.5-0.5B-Instruct} (500M, instruction-tuned, chat-templated) is the active default in production due to its speed and performance for lower range consumer machines; \texttt{Qwen2.5-1.5B-Instruct} (1.5B, same architecture) was included to test whether crossing into the 1B range improves temporal reasoning at the cost of latency; and \texttt{TinyLlama-1.1B-Chat-v1.0} (1.1B, a multi-turn chat-optimized model) was included to evaluate whether alternative chat architectures in the 1B range escape the structured-output ceiling. The encoder-decoder candidates are smaller T5 variants: \texttt{flan-t5-base} (250M, instruction-tuned via FLAN) and \texttt{LaMini-Flan-T5-248M} (248M, a distilled T5 trained for abstractive tasks). These were included to determine whether the seq2seq inductive bias offers advantages over decoder-only scaling for this task.

\texttt{all-MiniLM-L6-v2} (22M, a Sentence-BERT model) is evaluated not as a standalone grounder but as a retrieval augmentation component, deployed to broaden keyword coverage when embedded within the Curia pipeline. The model set thus captures both the architectural diversity of contemporary language models and the parameter-dependent performance cliffs that emerge under strict output constraints.

\section{Results}
\label{sec:results}

\subsection{Benchmark Design}

The temporal correctness suite evaluates all seven models on 72 labeled temporal queries from the Curia event corpus. Each query has a natural-language string, labeled keywords, expected temporal bounds (date and time fields), and one or more relevant event URLs. A secondary robustness dataset exists but is excluded from the primary analysis in this paper to focus depth on temporal grounding.

A broader \texttt{all-events} preset covering the full event catalog was available but exceeds 10 hours on consumer machines; we prioritize reproducibility and iteration speed within the temporal 72-query benchmark. All models receive the same structured prompt template with 4 temporal exemplars and 3 keyword exemplars, evaluated on the CUDA local backend for consistent hardware-independent latency comparison.

\subsection{Accuracy on the Temporal Dataset}

Table~\ref{table:temporal_accuracy} reports core extraction metrics across all seven models evaluated on the 72-query temporal benchmark. Parse success is the fraction of queries returning valid JSON. Date and time exact-match rates measure correctness of the extracted temporal bounds. Keyword F1 scores the overlap between predicted and labeled intent terms. All metrics are computed on the completed correctness suite.

Gemini (27B) achieves 91.7\% on all three extraction tasks (date, time, keywords). The local 8B model (Ollama llama3) is the only sub-Gemini candidate with non-zero temporal accuracy: 26.4\% date exact-match and 35.4\% time exact-match, with keyword F1 of 0.225 (37\% of Gemini's performance). Surprisingly, the 1.5B Qwen model matches or exceeds Gemini's temporal performance (91.7\% date exact-match, 93.8\% time exact-match) while nearly doubling the 8B latency. This suggests the 1.5B model contains sufficient capacity for temporal reasoning but the full Gemini prompt may underspecify the task. Below 1.5B, temporal accuracy degrades sharply: TinyLlama (1.1B) achieves 81.9\% on dates but only 52.1\% on times; Qwen 0.5B drops to 36.1\% date accuracy; and the T5 models (250M, 248M) achieve zero on both temporal fields.

\subsection{Parse Success and Semantic Failures}

Parse success across models reveals two distinct failure mechanisms. All seven models achieve >85\% parse success, indicating the output format is largely respected. However, models with low date/time exact-match rates produce syntactically valid JSON with semantically incorrect field values rather than malformed output.

Gemini (91.7\% parse) recovers from 8.3\% of queries via fallback date parsing. Ollama llama3 parses 100\% of queries but extracts incorrect temporal values; the model understands the output schema but struggles with temporal reasoning. The T5 models parse successfully but emit zero values in all temporal fields, suggesting they learned the JSON structure through FLAN instruction-tuning but lack the inductive capability to populate fields from the query semantics.

\subsection{Latency and Deployment Implications}

Inference latency ranges from 853 ms (LaMini) to 7,986 ms (Gemini), with intermediate models clustered by architecture. The fastest purely local model is flan-t5-base (860 ms), but it produces no correct temporal outputs and serves only as a speed baseline. Among models with usable accuracy, Ollama llama3 (3,551 ms) offers the best latency-accuracy tradeoff: 3.6\% slower than flan-t5-base but with 26.4\% date accuracy versus 0\%. Qwen 1.5B (2,930 ms) is the fastest among temporally-accurate models but exhibits higher per-query variance due to longer reasoning traces consuming CPU cycles non-uniformly.

\subsection{Key Findings}

\begin{enumerate}
    \item \textbf{Ollama 8B is the local Pareto frontier}: Retains 26.4\% date and 37\% keyword accuracy relative to Gemini at 3.6\,s latency. Smaller local models (Qwen 0.5B, TinyLlama) are slower, less accurate, or both.
    
    \item \textbf{Temporal reasoning has a hard scale-dependent ceiling below 1B parameters}: Qwen 1.5B matches Gemini on temporal tasks; Qwen 0.5B achieves 36\% on dates and 0\% on time. This is not a prompt-engineering gap but a capacity boundary for structured temporal reasoning at consumer CPU scales.
    
    \item \textbf{Keyword expansion is more size-resilient than temporal extraction}: Ollama llama3 achieves 37\% keyword F1 despite only 26.4\% date accuracy, confirming asymmetric degradation in extraction robustness.
\end{enumerate}

\section{Conclusion}
\label{sec:conclusion}

We present Curia-Lite, an efficiency study on model-size minimization for event query grounding. Our results confirm asymmetric degradation: under a consistent in-context learning protocol and fallback policy, keyword expansion retains partial function at 500M parameters while temporal structured output is essentially zero below the 27B scale on consumer CPU. A sub-1B structured-output ceiling is real and does not appear bridgeable by prompt engineering alone without quantization or GPU inference.

Key findings:
\begin{itemize}
    \item \textbf{Keyword expansion is more size-resilient}: At 500M parameters, keyword F1 retains 71\% of the 27B baseline. Temporal extraction retains 0\%.
    \item \textbf{A structured-output ceiling exists below 1B parameters on CPU}: No model smaller than 27B produces consistent parseable JSON with correct temporal arithmetic across the 72-query benchmark.
    \item \textbf{The 1.5B$\to$0.5B step is discontinuous for temporal reasoning}: The 1.5B model partially solves problems the 0.5B cannot, but at $14\times$ the latency, making it impractical without hardware acceleration.
    \item \textbf{Hybrid retrieval is an efficient supplement}: MiniLM at 22M parameters and $<$50\,ms augments keyword output for all model sizes with negligible overhead.
    \item \textbf{Fallbacks cannot close the gap}: The same \texttt{dateparser} fallback chain applies at every scale. It recovers explicit date strings but does not recover relative or compound temporal reasoning.
\end{itemize}

\subsection{Future Work}

\begin{itemize}
    \item \textbf{Quantized local models}: GGUF int4/int8 variants of Qwen2.5-1.5B and Qwen2.5-3B to improve the local accuracy-latency frontier without full-precision CPU inference.
    \item \textbf{Confidence-gated fallback}: Use token log-probabilities to decide when to trust LLM output versus fall back to rules, reducing unnecessary fallback invocations.
    \item \textbf{Recurring event queries}: Extend temporal extraction to handle recurring expressions (``every Monday'', ``weekly seminars'').
    \item \textbf{Expanded benchmark}: Add robustness (typo) and cross-domain variants to cover the full three-dataset suite (\textit{retrieval}, \textit{temporal}, \textit{robustness}).
\end{itemize}
